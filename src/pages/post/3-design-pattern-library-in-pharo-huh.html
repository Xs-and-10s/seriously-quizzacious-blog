PatternKit · Pharo 12 · All 23 GoF Patterns

Can a pattern library
be written for Pharo?

The answer is nuanced. 21 of the 23 GoF patterns can be encapsulated to some degree — reducing pattern boilerplate from scratch-every-time to subclass-and-fill-in-the-blanks. Two patterns fundamentally cannot be encapsulated, and understanding why teaches you something deep about OO design.

Tier 1 · Fully Encapsulated · Tier 2 · Partially Encapsulated · Tier 3 · Cannot Encapsulate
9

Fully Encapsulated

The library does all the structural wiring. You subclass, pass a block, or just use the class directly. Zero boilerplate to rewrite.

SingletonCommand+UndoChain FlyweightObserverState Machine MementoDecoratorProxy
12

Partially Encapsulated

The library eliminates structural boilerplate. Domain-specific logic still requires subclassing, but the scaffolding is pre-built.

2

Cannot Encapsulate

Not a limitation of Pharo — a fundamental truth about these patterns. A library class would add nothing beyond Object.

Install the library — use it like this

Load PatternKit from a Metacello baseline. Every class is prefixed PK. For fully-encapsulated patterns, you may never need to write a single pattern class from scratch again. For partially-encapsulated ones, you subclass one of the PK* base classes and override only what's domain-specific.

The library ships with every pattern as a working example in its test suite — the tests are the best usage documentation.

Tier 1

Fully Encapsulated

These patterns can be reduced to zero structural boilerplate. Pass blocks for variable behavior, or simply subclass with no overrides required for the common case.

01 Singleton Creational Fully Encapsulated ▼

Key trick: class-instance variables (not class variables). Each subclass of PKSingleton gets its own independent uniqueInstance slot on its metaclass — not shared with siblings.

Library · PKSingleton
PatternKit-Creational 
Object subclass: #PKSingleton
    instanceVariableNames: ''
    classVariableNames:    ''
    package: 'PatternKit-Creational'

"Class-instance var — one per subclass's metaclass,
 NOT shared across the hierarchy like classVariableNames"
PKSingleton class
    instanceVariableNames: 'uniqueInstance'

PKSingleton class >> uniqueInstance
    uniqueInstance isNil ifTrue: [
        uniqueInstance := super new initialize ].
    ^ uniqueInstance

"Block direct new calls at the subclass level"
PKSingleton class >> new
    ^ self error: 'Use ' , self name , ' uniqueInstance'

"Essential for test isolation"
PKSingleton class >> resetForTesting
    uniqueInstance := nil
Usage — zero boilerplate
MyApp 
"Subclass — that's it. No class var, no guard logic."
PKSingleton subclass: #AppConfig
    instanceVariableNames: 'settings'
    classVariableNames:    ''
    package: 'MyApp'

AppConfig >> initialize
    settings := Dictionary new.
    settings at: #host put: 'localhost'.
    settings at: #port put: 8080

AppConfig >> at: k        ^ settings at: k
AppConfig >> at: k put: v  settings at: k put: v

"Usage"
AppConfig uniqueInstance at: #host.
"=> 'localhost'"
AppConfig uniqueInstance == AppConfig uniqueInstance.
"=> true"
AppConfig new.
"=> Error: Use AppConfig uniqueInstance"
02 Command + Undo History Behavioral Fully Encapsulated ▼

Two classes: PKCommand parameterized with execute/undo blocks, and PKCommandHistory managing a bounded undo/redo stack. No subclassing ever needed.

Library · PKCommand + PKCommandHistory
PatternKit-Behavioral 
Object subclass: #PKCommand
    instanceVariableNames: 'execBlock undoBlock label'
    classVariableNames: ''
    package: 'PatternKit-Behavioral'

PKCommand class >> execute: eBlk undo: uBlk
    ^ self new
        pvtExec: eBlk undo: uBlk;
        yourself

PKCommand class >> execute: eBlk undo: uBlk label: s
    ^ (self execute: eBlk undo: uBlk) label: s; yourself

PKCommand >> pvtExec: e undo: u  execBlock := e. undoBlock := u
PKCommand >> label: s            label := s
PKCommand >> label               ^ label ifNil: ['Command']
PKCommand >> execute             execBlock value
PKCommand >> undo
    undoBlock isNil ifTrue: [
        ^ self error: self label , ' has no undo' ].
    undoBlock value
PKCommand >> isUndoable  ^ undoBlock notNil

"────────────────────────────"
Object subclass: #PKCommandHistory
    instanceVariableNames: 'undoStack redoStack limit'
    classVariableNames: ''
    package: 'PatternKit-Behavioral'

PKCommandHistory >> initialize
    undoStack := OrderedCollection new.
    redoStack := OrderedCollection new.
    limit := 100

PKCommandHistory >> limit: n   limit := n

PKCommandHistory >> execute: cmd
    cmd execute.
    undoStack addLast: cmd.
    undoStack size > limit ifTrue: [undoStack removeFirst].
    redoStack removeAll

PKCommandHistory >> undo
    undoStack isEmpty ifTrue: [^ self].
    | cmd | cmd := undoStack removeLast.
    cmd undo.
    redoStack addLast: cmd. ^ cmd

PKCommandHistory >> redo
    redoStack isEmpty ifTrue: [^ self].
    | cmd | cmd := redoStack removeLast.
    cmd execute.
    undoStack addLast: cmd. ^ cmd

PKCommandHistory >> canUndo    ^ undoStack notEmpty
PKCommandHistory >> canRedo    ^ redoStack notEmpty
PKCommandHistory >> undoLabels ^ undoStack collect: [:c|c label]
PKCommandHistory >> clear
    undoStack removeAll. redoStack removeAll
Usage — no subclassing whatsoever
MyApp 
"Wire up a full undo/redo system with two objects
 and zero pattern classes of your own"
| buf history |
buf     := ''.
history := PKCommandHistory new.

history execute: (PKCommand
    execute: [buf := buf , 'Hello']
    undo:    [buf := buf copyFrom: 1 to: buf size - 5]
    label:   'Insert Hello').

history execute: (PKCommand
    execute: [buf := buf , ' World']
    undo:    [buf := buf copyFrom: 1 to: buf size - 6]
    label:   'Insert World').

buf.               "=> 'Hello World'"
history undoLabels. "=> #('Insert Hello' 'Insert World')"
history undo.      buf. "=> 'Hello'"
history undo.      buf. "=> ''"
history redo.      buf. "=> 'Hello'"
history canRedo.        "=> true"
03 Chain of Responsibility Behavioral Fully Encapsulated ▼

PKHandler can be used two ways: (1) pass blocks — build an entire handler chain with zero subclasses; (2) subclass for handlers with complex state. Both share the same chainTo: fluent API.

Library · PKHandler
PatternKit-Behavioral 
Object subclass: #PKHandler
    instanceVariableNames: 'next canBlock doBlock'
    classVariableNames: ''
    package: 'PatternKit-Behavioral'

"Block constructor — zero subclassing needed"
PKHandler class >> canHandle: cBlk then: dBlk
    ^ self new
        pvtCan: cBlk do: dBlk;
        yourself

PKHandler >> pvtCan: c do: d  canBlock := c. doBlock := d

"Fluent chain building — returns next for chaining"
PKHandler >> chainTo: h   next := h. ^ h
PKHandler >> next: h        next := h

PKHandler >> handle: req
    (self canHandle: req)
        ifTrue:  [^ self process: req]
        ifFalse: [^ next
            ifNotNil: [next handle: req]
            ifNil:    [self unhandled: req]]

"Override in subclasses, or the block is used"
PKHandler >> canHandle: req
    ^ canBlock notNil
        ifTrue:  [canBlock value: req]
        ifFalse: [self subclassResponsibility]

PKHandler >> process: req
    ^ doBlock notNil
        ifTrue:  [doBlock value: req]
        ifFalse: [self subclassResponsibility]

"Terminal handler hook — override to log/error"
PKHandler >> unhandled: req    "default: silent"
Usage — block form, zero subclasses
MyApp 
"Entire approval chain with no pattern subclasses"
| chain |
chain := (PKHandler
    canHandle: [:r| r amount <= 500]
    then:       [:r| Transcript showCr:
        'Team Lead approved $',r amount printString]).

(chain chainTo: (PKHandler
    canHandle: [:r| r amount <= 5000]
    then:       [:r| Transcript showCr:
        'Manager approved $',r amount printString]))
chainTo: (PKHandler
    canHandle: [:r| true]
    then:       [:r| Transcript showCr:
        'CFO approved $',r amount printString]).

chain handle: req100.   "=> 'Team Lead...'"
chain handle: req2000.  "=> 'Manager...'"
chain handle: req50k.   "=> 'CFO...'"
04 State Machine Behavioral Fully Encapsulated ▼

PKStateMachine delegates all domain messages to the current state via doesNotUnderstand:. PKState provides transitionTo:, and onEnter/onExit hooks. You only write the state classes and their transition logic.

Library · PKStateMachine + PKState
PatternKit-Behavioral 
Object subclass: #PKState
    instanceVariableNames: 'context'
    classVariableNames: ''
    package: 'PatternKit-Behavioral'

PKState >> context: c   context := c
PKState >> context       ^ context

"The key method — state classes call this to transition"
PKState >> transitionTo: stateClass
    context pvtTransitionTo: stateClass

"Lifecycle hooks — override as needed"
PKState >> onEnter    "called when entering state"
PKState >> onExit     "called when leaving state"

"Default for unhandled events: raise a clear error"
PKState >> doesNotUnderstand: msg
    self error: self class name , ' cannot handle: ' , msg selector

"────────────────────────────"
Object subclass: #PKStateMachine
    instanceVariableNames: 'state'
    classVariableNames: ''
    package: 'PatternKit-Behavioral'

PKStateMachine class >> startingIn: stateClass
    ^ self new pvtStartIn: stateClass; yourself

PKStateMachine >> pvtStartIn: stateClass
    state := stateClass new context: self; yourself.
    state onEnter

PKStateMachine >> pvtTransitionTo: stateClass
    state onExit.
    state := stateClass new context: self; yourself.
    state onEnter

PKStateMachine >> state      ^ state
PKStateMachine >> stateName  ^ state class name

"All domain messages auto-delegate to the current state"
PKStateMachine >> doesNotUnderstand: msg
    ^ msg sendTo: state

05 Memento Behavioral Fully Encapsulated ▼

PKOriginator uses introspection to capture/restore all instance variables generically. PKCaretaker manages the undo/redo stacks. For most objects, zero overrides needed. Override save/restore: only for sparse (delta) mementos.

Library · PKOriginator + PKCaretaker
PatternKit-Behavioral 
Object subclass: #PKOriginator
    instanceVariableNames: ''
    classVariableNames: ''
    package: 'PatternKit-Behavioral'

"Default: full deep copy — override for sparse mementos"
PKOriginator >> save
    ^ self deepCopy

"Default: restore all inst vars by reflection"
PKOriginator >> restore: memento
    self class allInstVarNames doWithIndex: [:n :i|
        self instVarAt: i put: (memento instVarAt: i)]

"────────────────────────────"
Object subclass: #PKCaretaker
    instanceVariableNames: 'orig undo redo'
    classVariableNames: ''
    package: 'PatternKit-Behavioral'

PKCaretaker class >> for: anOriginator
    ^ self new pvtFor: anOriginator; yourself

PKCaretaker >> pvtFor: o
    orig := o. undo := OrderedCollection new.
    redo := OrderedCollection new

PKCaretaker >> checkpoint
    undo addLast: orig save. redo removeAll

PKCaretaker >> undo
    undo isEmpty ifTrue: [^ self].
    redo addLast: orig save.
    orig restore: undo removeLast

PKCaretaker >> redo
    redo isEmpty ifTrue: [^ self].
    undo addLast: orig save.
    orig restore: redo removeLast

PKCaretaker >> canUndo   ^ undo notEmpty
PKCaretaker >> canRedo   ^ redo notEmpty
PKCaretaker >> depth     ^ undo size
Usage — subclass PKOriginator, done
MyApp

06 Observer Behavioral Already in Pharo ▼

Observer is built into the image twice: the legacy Model/changed:/update: dependency mechanism, and the modern typed Announcer framework. PKObservable is a Trait that lazily adds an Announcer to any class, so you don't have to subclass Model.

Library · PKObservable Trait
PatternKit-Behavioral 
"Trait — mix into any class, not just Model subclasses"
Trait named: #PKObservable
    instanceVariableNames: 'pkAnnouncer'
    package: 'PatternKit-Behavioral'

"Lazy init — announcer created only on first use"
PKObservable >> announcer
    pkAnnouncer isNil ifTrue: [
        pkAnnouncer := Announcer new ].
    ^ pkAnnouncer

"Subscribe by announcement class + block"
PKObservable >> on: annClass do: aBlock
    ^ self announcer on: annClass do: aBlock

"Subscribe with message-send (avoids retain-cycles)"
PKObservable >> on: annClass send: msg to: receiver
    ^ self announcer on: annClass send: msg to: receiver

"Fire an announcement"
PKObservable >> announce: ann
    pkAnnouncer isNil ifTrue: [^ self]. "no-op if no subs"
    pkAnnouncer announce: ann

PKObservable >> unsubscribe: sub
    pkAnnouncer isNil ifTrue: [^ self].
    pkAnnouncer unsubscribe: sub

PKObservable >> hasObservers
    ^ pkAnnouncer notNil

PKDecorator subclasses ProtoObject and uses doesNotUnderstand: to auto-delegate every unrecognized message to the wrapped component. You only override the specific methods you want to intercept — everything else passes through automatically.

    instanceVariableNames: 'selector beforeBlock afterBlock'
    classVariableNames: ''
    package: 'PatternKit-Structural'

PKAroundDecorator class >>
  wrapping: c intercept: sel before: bBlk after: aBlk
    ^ (self wrapping: c)
        pvtSel: sel before: bBlk after: aBlk; yourself

    selector := s. beforeBlock := b. afterBlock := a

    msg selector = selector ifTrue: [
        beforeBlock value: msg arguments.
        | r | r := msg sendTo: self component.

Three concrete proxy types ship with PatternKit: PKLoggingProxy, PKVirtualProxy (lazy loader), and PKProtectionProxy. Custom proxies subclass PKProxy and override beforeMessage:/afterMessage:result:.

09 Flyweight Factory Structural Fully Encapsulated ▼

PKFlyweightFactory provides the caching registry. You supply a key extraction block and the flyweight class. The factory handles all the "create if not cached, return cached if present" logic automatically.

Library · PKFlyweightFactory
PatternKit-Structural 
Object subclass: #PKFlyweightFactory
    instanceVariableNames: 'cache keyBlock creatorBlock'
    classVariableNames: ''
    package: 'PatternKit-Structural'

"keyBlock: extracts cache key from args
 creatorBlock: creates a new flyweight from args"
PKFlyweightFactory class >>
  keyedBy: keyBlk creating: creatorBlk
    ^ self new
        pvtKey: keyBlk create: creatorBlk;
        yourself

PKFlyweightFactory >> initialize   cache := Dictionary new
PKFlyweightFactory >> pvtKey: k create: c
    keyBlock := k. creatorBlock := c

PKFlyweightFactory >> get: args
    | key |
    key := keyBlock value: args.
    ^ cache at: key ifAbsent: [
        | fw |
        fw := creatorBlock value: args.
        cache at: key put: fw.
        fw]

PKFlyweightFactory >> cacheSize   ^ cache size
PKFlyweightFactory >> purge        cache := Dictionary new
PKFlyweightFactory >> cachedKeys   ^ cache keys
Usage — define key + creator, done
MyApp

Tier 3

Cannot Be Encapsulated

Not a limitation of Pharo. These two patterns cannot be encapsulated in any OO language. The reason reveals something fundamental about what patterns are.

⊘ Facade

Consider what a PKFacade library class would look like. It would need to simplify your subsystem, but it knows nothing about your subsystem. The only option is:

Attempted PKFacade 
"The honest attempt:"
Object subclass: #PKFacade
    instanceVariableNames: ''
    classVariableNames: ''
    package: 'PatternKit-Structural'

"That's it. That's the whole class. It IS Object."
"Nothing useful can be added without knowing what subsystem is being wrapped."

A PKFacade superclass would add literally zero functionality beyond Object. The Facade pattern's entire value IS the domain-specific simplification you write inside it. The simplification cannot be parameterized because it presupposes knowing the structure of your subsystem — which a general-purpose library cannot know.

What PatternKit can provide here is a marker and a convention: inherit from PKFacade to signal architectural intent to your team. But that is documentation, not encapsulation.

The insight: Facade is a pattern about what you write — simple methods covering a complex subsystem. It is not a structural arrangement of objects. There is nothing structural to abstract. Any pattern whose "implementation" IS just writing methods in a class is immune to library encapsulation.

⊘ Template Method

This is the more profound of the two. Template Method says: "write a method that calls other methods, some of which are abstract." Consider what you would need from a library to avoid writing this yourself:

Attempted PKTemplateMethod 
"What would the library class need to provide?
 The template method IS just a normal method."

Object subclass: #PKTemplateMethod
    instanceVariableNames: ''
    classVariableNames: ''
    package: 'PatternKit-Behavioral'

"The 'template' IS a method body. You have to write it."
PKTemplateMethod >> run
    self setup.
    self doWork.
    self teardown

"The 'primitives' ARE subclassResponsibility — already in Pharo."
PKTemplateMethod >> setup    "hook"
PKTemplateMethod >> doWork   ^ self subclassResponsibility
PKTemplateMethod >> teardown "hook"

"But this only encapsulates THIS specific algorithm skeleton.
 Every different skeleton needs its own template method.
 A library can never write your algorithm skeleton for you."

The issue is that every Template Method is a different algorithm with a different skeleton. A library can provide one skeleton — but then it IS just one specific algorithm, not a general-purpose pattern.

More deeply: Template Method is not a structural arrangement of objects at all. It is a description of how OO inheritance already works. Every time you write a method in a superclass that calls overridable methods, you have implemented Template Method. The "library" for Template Method is Pharo's object model itself — subclassResponsibility is its primitive marker. It shipped with the first Smalltalk.

The insight: Some GoF patterns are structural arrangements of objects that can be abstracted into reusable base classes (Strategy, Command, Chain, State). Others are simply descriptions of how OOP already works (Template Method) or descriptions of purposeful writing (Facade). The former are encapsulateable; the latter never will be, in any language.

This is why the GoF book itself says about Template Method: "template methods are so fundamental they can be found in almost every abstract class." It's not a design choice you make — it's a consequence of using inheritance at all.

PatternKit — what it looks like in practice

Loading PatternKit gives you 9 fully-encapsulated pattern objects you use directly, 12 base classes you subclass with only domain-logic overrides, and 2 marker conventions (Facade, Template Method) that document intent but add no code.

For a typical Pharo application, the patterns you reach for most (Observer, Command, Strategy, Iterator, State, Memento, Chain) are all in Tier 1 — fully encapsulated, zero structural boilerplate. The patterns requiring genuine subclassing (Composite, Visitor, Interpreter, Abstract Factory) are the ones where domain structure is intrinsic to the pattern's value, and no library can know that structure for you.

The honest verdict: Yes, a PatternKit library is worth writing and using. But the patterns you'll most want to avoid rewriting from scratch — Command, Observer, State Machine, Memento — are precisely the ones PatternKit handles completely. The patterns that still need your code are the ones that should need your code: they're encoding your domain model, and that's not something a library can do for you.
